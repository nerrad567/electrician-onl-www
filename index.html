usb-verifier — deep dive

A PowerShell-based verification engine for Windows install USBs, designed to catch subtle tampering and give operators a clear verdict.

Problem & threat model

  • Standard ‚Äúcopy ISO to USB‚Äù workflows make it easy for hidden malware or modified binaries to sneak in unnoticed.
  • Goal: quickly tell whether a USB still faithfully represents a known-good ISO, including the contents of WIM/ESD/SWM images.

Engine design

  • Environment validation phase: checks tools, paths, and that the ISO/USB are in a safe state to analyse.
  • Root (surface) comparison: walks ISO vs USB file trees, classifies files as matched, modified, extra, or missing, and summarises results.
  • Deep image analysis:
    • Mounts WIM/ESD and reassembles split SWM sets where needed.
    • Hashes and inspects internal files, signatures, and key resources.
    • Treats boot/install images differently from generic content to keep reports focused.

Reporting & operations

  • Structured log output with timestamps and phase tags, suitable for use in tickets or security reviews.
  • Final human-readable summary section: integrity verdict, counts of modified/extra/missing files, and any key findings.
  • Built as a single script for now (~2k lines), but structured for future refactors into modules.

Non-invasive 100 A mains incomer monitoring with an ESP32 edge node and a containerised backend (MQTT → time-series DB → FastAPI UI).

100 A CT hardware spec

  • CT sensor: 100 A split-core CT, SCT-013-000-class (100 A : 50 mA).
  • Ratio: 100 A primary → 50 mA secondary (2000:1).
  • Burden resistor: target ‚âà1.1 V RMS at 100 A across the CT, so R ‚âà 1.1 V / 0.05 A ‚âà 22 Œ©.
  • Burden power: at 50 mA, P = I¬≤R ‚âà 0.055 W, so a 0.25 W metal film resistor has margin.
  • Bias network: 2 √ó 10 kŒ© forming a divider from 3.3 V to GND, midpoint at ~1.65 V, decoupled with 100 nF–1 ¬µF to ground.
  • The CT/burden voltage is referenced to this mid-rail so the AC waveform stays within the 0 V to 3.3 V ADC range.
  • MCU: ESP32 DevKit-style board (Wi-Fi, onboard USB-serial) using one ADC channel for the CT input, with spare GPIOs for future clamps or status LEDs.
  • Power: development via USB 5 V; production via a small DIN-rail or panel PSU feeding the ESP32 inside an enclosure.

Planned firmware behaviour

  • On boot, load a sensor profile (100 A CT, 22 Œ© burden, 230 V mains) from config.
  • Bring up Wi-Fi and connect to the MQTT broker.
  • Sample the ADC over one or more mains cycles, either computing RMS current on the ESP32 or sending aggregated sample stats upstream.
  • Build a JSON payload with timestamp, node ID, current, power, and any health flags, and publish to a topic like clamps/main_incomer using QoS 1.
  • Implement basic resilience: small local buffer for short outages and reconnect with backoff on Wi-Fi/MQTT failure.

Planned backend behaviour

  • MQTT broker receives telemetry and enforces topic-level ACLs (clamps/<site>/<circuit>).
  • A FastAPI-based worker subscribes to clamp topics, validates payloads, performs any remaining RMS / power calculations, and writes points to a time-series database.
  • HTTP endpoints like /api/history and /api/stats serve historical data and daily/weekly summaries.
  • A lightweight dashboard (served by FastAPI) uses WebSockets for live-updating charts (Chart.js) and an event log (spikes, drops, gaps).

Security profile

  • Baseline: MQTT over TLS for this personal / portfolio deployment (energy-only, read-only).
  • Optional hardened mode: MQTT over QUIC (TLS 1.3) plus WireGuard as an overlay VPN so broker ports are only reachable inside a mesh of authenticated peers.
  • The design keeps the default deployment lean, but allows the same node and backend to be reused in networks with stricter security requirements.
